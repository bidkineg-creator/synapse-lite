#### Bug #1 — Offer Upload يرجع 403 ← الأخطر

**الوصف:**  
Flutter يسمح لـ 4 أدوار برفع عروض: `merchant, company, supplier, institution`  
Backend يسمح لـ 2 فقط: `merchant, company`

**النتيجة:** التاجر يضغط رفع عرض → يحصل على 403 بدون شرح → يعتقد المنصة معطلة.

**الإصلاح المطلوب:**
```
الخيار أ: وحّد backend مع Flutter — أضف supplier + institution لـ whitelist
الخيار ب: أخفِ زر الرفع في Flutter لو الدور مش مسموح له
```
أيضاً: رفع العرض مقيّد بـ:
- user_type != individual
- account_status != pending
- وجود grocery_settings (للسوبرماركت)

هذه القيود يجب أن تعطي رسائل واضحة لا 403.

---

#### Bug #2 — حفظ التسجيل يقول "اكتمل" قبل الحفظ الفعلي

**الوصف:**  
الخطوة 4 في التسجيل تعرض "تم الحفظ / اكتمل" — لكن الحفظ الحقيقي يحدث عند الضغط على "العودة للرئيسية".  
لو المستخدم أغلق التطبيق بعد "اكتمل" ← التسجيل ناقص.

**الإصلاح المطلوب:**
```
1. لا تعرض "اكتمل" إلا بعد success: true من السيرفر
2. افصل رسالتين مختلفتين:
   ✅ "تم حفظ بياناتك"
   ⏳ "طلبك قيد المراجعة (24-48 ساعة)"
3. RequestGuard الموجود (3 ثواني) يبقى — يمنع الضغط المزدوج
```

---

#### Bug #3 — صورة البروفايل: خطأ مضلل بعد رفع ناجح

**الوصف:**  
الصورة تُرفع بنجاح لكن عملية my-tree تفشل بعدها → يظهر خطأ كأن الرفع فشل.

**الإصلاح المطلوب:**
```
افصل الـ error handling:
- لو الصورة اترفعت: اعرض ✅ "تم تحديث الصورة"
- لو my-tree فشل: سجّل الخطأ داخلياً بدون إزعاج المستخدم
```

## ثالثاً: القرارات التقنية المحسومة

| القرار | الحكم | السبب |
|--------|-------|-------|
| Flutter أم Next.js؟ | Flutter — الموجود | 70% من الشغل موجود. Rebuild = أشهر ضائعة |
| Bull + Redis للمزايدة؟ | لا — في الوقت الحالي | auto_bid_service.js شغال. أضف Bull/Redis لو وصلت 1000+ مزاد متزامن |
| MinIO للصور؟ | لا — Filesystem يكفي | uploads/ موجود. MinIO تحتاجه بعد 500+ صورة |
| قاعدة بيانات جديدة؟ | لا | reverse-engineer الـ schema الموجود وطوّر عليه |
| Parametric DashboardShell؟ | لا | Flutter عنده widget composition — /role-dashboard + GET /dashboard/me يكفي |
